# Claudio 在线电台 Agent 跟进计划

## 1. 背景

Claudio 已经具备节目计划、TTS 串场、双 audio 预加载、网易云播放 URL 刷新、unblock fallback 和基础节目审计。当前下一阶段重点不是启用本地曲库，而是在在线音源链路上继续提升稳定性，并让 Claudio 能根据用户反馈进行有限探索。

## 2. 目标

最终目标是把 Claudio 做成一个稳定、可连续收听、会观察反馈并适度探索新歌的在线 AI 电台 agent。

目标效果：

- 播放源稳定，不再长期出现“进度条在走但没声音”或旧 URL 过期后继续播放。
- 后端能主动维护当前歌和后续歌曲的播放 URL，而不是只在出错后补救。
- 前端能看到当前播放源、fallback、URL 过期时间和最近错误，便于实时排查。
- 用户反馈、跳过、收藏、听完等信号能影响后续节目。
- Claudio 能按比例探索歌单外的新歌，但探索必须可播放、可解释、可收回。
- DJ 话术保持自然，不装深沉，不过度抒情，不频繁打断音乐。

## 3. 范围

本阶段做：

- 在线播放 URL 租约刷新。
- 播放源健康状态和诊断面板。
- fallback 触发条件加强。
- 显性和隐性反馈记录。
- 动态口味画像。
- Discovery 候选池。
- 节目探索比例控制。
- 20 分钟实听验收。

本阶段不做：

- 不启用本地曲库。
- 不做复杂账号体系。
- 不做大规模机器学习推荐模型。
- 不把不稳定或高风险音源作为主链路。
- 不让 LLM 直接凭空决定最终可播放歌曲。

## 4. 关键规则

1. 音乐连续性优先于 DJ 文案完整性。
2. 队列里的播放 URL 必须有有效期和健康状态。
3. 拿到 URL 不等于可播放，必须能返回可识别音频。
4. 当前歌和下一首必须优先维护，其次才是更远的队列。
5. fallback 只解决播放链接问题，不改变歌曲身份。
6. 用户反馈优先影响后续队列，不要频繁打断当前歌曲。
7. 探索必须有比例控制，不能把节目变成随机歌单。
8. DJ 说话要短、具体、有判断，不写散文。
9. 诊断面板不能暴露 Cookie、密钥、真实私密配置。
