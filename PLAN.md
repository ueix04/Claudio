# Claudio AI 电台改善计划

## 1. 背景

Claudio 目前更像“AI 推荐歌曲 + 普通播放器”，还不够像一个持续运行的 AI 电台 agent。

当前主要问题：

- Agent 调度不连续：LLM 多数在触发推荐、聊天或切歌前才调用，缺少提前规划和后台准备。
- 切歌准备来不及：后端会提前准备下一首串场和 TTS，但前端没有真正预加载下一首音乐音频。
- 聊天不像真人：普通问题容易走泛化回复，缺少先理解用户意图再决定动作的流程。
- 天气使用过度：天气上下文被频繁塞给 LLM，导致回复经常强行提天气。
- 网易云音源不稳定：当前依赖网易云临时播放 URL，URL 过期、502、版权限制或 Cookie 问题都会直接影响播放。

## 2. 目标

最终目标是把 Claudio 从“点歌播放器”升级为“持续运行的 AI 电台 agent”。

目标效果：

- Claudio 能提前规划一段节目，而不是临时等切歌时才思考。
- 当前歌曲播放时，后台已经准备好后续 2 到 3 首歌、播放链接和必要串场。
- 用户聊天时，Claudio 能先判断意图，再自然回答或调整节目。
- 天气只在用户询问、节目开场或确实影响氛围时出现。
- 切歌更顺滑，正常网络下没有明显停顿。
- 网易云失败时可以刷新、重试、降级或切换备用音源。
- 后续可以逐步支持本地音乐库，降低对第三方音源的依赖。

## 3. 范围

### 做什么

- 重构 AI 电台调度逻辑，引入持续节目规划。
- 增加聊天意图识别，避免所有聊天都走同一个回复模板。
- 调整天气上下文使用规则。
- 抽象音乐源适配层，替换当前强绑定网易云临时 URL 的方式。
- 增加音源重试、刷新、fallback 和失败跳过机制。
- 增加前端下一首音频预加载、淡入淡出和 DJ 说话时的音乐压低。

### 暂不做什么

- 不一次性重写整个前后端。
- 不把未知来源的音源插件直接放进生产链路。
- 不把第三方音源当作稳定商业服务承诺。
- 不为了“更像 agent”而无意义地高频调用 LLM。

## 4. 关键规则

1. 音乐连续性优先于 AI 文案完整性。
2. 切歌不能等待 LLM 或 TTS。
3. 队列里不要长期保存网易云临时播放 URL。
4. 播放前或预取前再刷新真实播放链接。
5. 天气不是默认聊天上下文，只在相关场景使用。
6. 用户普通聊天要先识别意图，再决定是否回答、换歌、查天气或重排节目。
7. 网易云继续可作为口味来源，但不应作为唯一播放保障。
8. 第三方音源必须封装在适配层里，便于替换和禁用。
9. 外部项目接入前需要再次确认维护状态、使用方式和风险。
10. 敏感信息仍只放在 `.env`，不得写入代码、日志或文档。

- [x] 普通聊天不默认携带天气上下文。
- [x] 只有用户明确问天气时，才调用天气接口并回答天气。
- [x] 节目开场可以自然提一次天气，但后续不要重复提。
- [x] 用户说“换安静一点”“来点更燃的”时，不必立即中断当前歌，优先重排下一组节目。

## 阶段三：重构音源架构（已完成：2026-06-08）

目标：解除对单一网易云临时 URL 的依赖。

事项：

- [x] 新增 `MusicSourceAdapter` 抽象。
- [x] 适配层至少包含：搜索、获取详情、解析播放链接、刷新链接、健康检查。
- [x] 队列中以歌曲 ID、标题、歌手、专辑、时长、来源作为稳定身份，临时 URL 只作为带过期时间的运行时缓存。
- [x] 播放前或预取前动态解析真实播放 URL。
- [x] 给解析结果加过期时间。
- [x] 音源失败时返回结构化错误，例如 `url_expired`、`source_502`、`unplayable`。
- [x] 接入主源和备用源解析链路：主源失败时，用同一个网易云歌曲 ID 交给备用源解析。
- [x] 记录当前播放 URL 的解析来源，区分歌曲身份来源和 URL 来源。

推荐接入顺序：

- [x] 主源：当前保留 `netease_legacy` 适配器作为主源；NeteaseCloudMusicApiEnhanced 先作为替换候选，不在本轮强行引入。
- [x] 备用源：接入 UnblockNeteaseMusic/server，仅作为失败 fallback，不作为唯一主源。
- [x] 长期稳定源：本地音乐库基础接入，用网易云做口味画像时可优先匹配本地文件做可靠播放。

本地音乐库实现记录：

- [x] 新增 `local_library` 音源适配器，支持递归扫描本地音频目录、搜索、获取详情、解析播放链接和健康检查。
- [x] 本地曲目使用稳定 `sourceTrackId`，队列只保存本地曲目身份和 `/api/audio/local/:sourceTrackId` 播放入口，不保存真实磁盘路径。
- [x] 配置 `LOCAL_MUSIC_DIRS` 后，默认解析顺序变为本地音乐库优先，未命中再走 `netease_legacy`。
- [x] 网易云口味候选曲命中时，会先用标题和歌手尝试匹配本地文件，匹配失败再回到网易云 ID 解析。
- [x] 前端识别项目内 `/api/audio/...` 播放地址，避免把本地音频接口错误包进远程音频代理。
- [x] 本地音乐库状态和重扫 API 已接入：`GET /api/music-sources/local-library`、`POST /api/music-sources/local-library/rescan`。
- [x] 本地音乐库摘要已进入节目编排上下文；没有网易云快照时，配置了本地曲库也可以作为开场节目依据。
- [x] 前端已展示本地音乐库状态：首屏 signal 显示本地曲目数，TASTE 面板显示启用状态、目录可用数、扫描时间、样例曲目和重扫入口。

风险记录：

- [x] UnblockNeteaseMusic/server 只作为可禁用 fallback；可通过 `UNBLOCK_NETEASE_ENABLED=false` 关闭，通过 `UNBLOCK_NETEASE_SOURCES` 调整匹配源。
- [x] 默认备用匹配源使用 `kugou,bodian,migu`，不默认依赖 `ytdlp` 外部二进制。
- [x] 本地音乐库当前先按文件名解析 `Artist - Title`，暂不读取音频元数据；真实曲库整理质量会影响搜索命中率。
- [x] 本地音乐库状态 API 不返回真实磁盘路径，只返回启用状态、目录数量、曲目数量和少量曲目样例。
- [x] 本地音乐库暂不提供前端目录导入界面；通过 `.env` 的 `LOCAL_MUSIC_DIRS` 配置目录，重扫已可在前端 TASTE 面板触发。

验收：

- [x] 队列不再依赖旧的网易云临时 URL。
- [x] 旧 URL 过期不会导致整条播放链路卡死。
- [x] 音源失败时可以刷新、重试或自动跳过。
- [x] 配置本地音乐目录后，可以优先返回本地文件播放地址，并通过后端安全路由播放。
- [x] 配置本地音乐目录后，LLM 候选上下文包含本地可播放曲目，选中后会解析为 `local_library` 播放源。
- [x] 用户能在前端看到本地曲库是否就绪，并手动触发本地曲库重扫。

## 6. 技术拆分建议

建议优先新增或调整这些模块：

- `backend/src/agent-router.ts`：聊天意图识别和动作分发。
- `backend/src/radio-session.ts`：节目会话、节目计划、后台准备状态。
- `backend/src/music-sources/`：音乐源适配层。
- `backend/src/music-sources/netease-enhanced.ts`：网易云增强 API 适配。
- `backend/src/music-sources/unblock-netease.ts`：备用解析适配。
- `backend/src/playback-planner.ts`：下一首、串场、fallback 准备。
- `frontend/src/hooks/useRadioPlayback.ts`：双 audio、预加载、淡入淡出、错误恢复。

保持现有模块逐步迁移：

- `pipeline.ts` 先保留，但把“生成单次推荐”逐步迁到 Radio Session。
- `netease.ts` 先作为旧适配器保留，等新适配器稳定后再替换。
- `useWebSocket.ts` 先拆播放相关逻辑，避免继续膨胀。

## 7. 验收标准

整体完成后，需要满足：

- Claudio 能生成并维护一段完整节目，而不是只临时推荐单首歌。
- 后续 2 到 3 首歌始终提前准备。
- 聊天回复能根据用户问题自然回答。
- 天气只在相关时出现。
- 切歌正常情况下无明显卡顿。
- 网易云 502 不会直接导致播放中断。
- 音源失败有自动刷新、fallback 或跳过。
- DJ 说话时音乐自动压低。
